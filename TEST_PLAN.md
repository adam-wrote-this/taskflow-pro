# TaskFlow Pro 测试计划

## 1. 目标

本计划用于保证 TaskFlow Pro 在以下方面的稳定性与可回归性：
- 认证与会话可靠（登录、注册、退出、会话续期）
- 团队/项目/任务核心流程可用（创建、编辑、拖拽、成员管理）
- 多语言与设置功能正确（中文/英文切换、账户信息修改）
- Supabase 数据权限与枚举一致性可验证（RLS、task_status）

## 2. 测试范围

### 2.1 In Scope（本期纳入）
- Web 前端页面和交互（Next.js App Router）
- Supabase 客户端/服务端调用路径
- 中间件会话刷新与受保护路由跳转
- SQL 迁移相关的关键行为回归
- 基础非功能验证（性能烟测、安全基线）

### 2.2 Out of Scope（本期不纳入）
- 原生移动端
- 大规模压测（并发压测）
- 复杂灾备演练（跨区域容灾）

## 3. 系统与风险概览

### 3.1 关键业务链路
- 认证：注册、登录、退出
- 协作：创建团队 -> 邀请成员 -> 创建项目 -> 创建任务 -> 拖拽任务状态
- 维护：修改个人资料、修改密码、语言切换

### 3.2 高风险点（优先验证）
- 看板拖拽后 position 与 status 的一致性（前端状态与数据库最终一致）
- 团队成员权限边界（owner/admin/member 的操作差异）
- RLS 规则与团队数据隔离（防越权读写）
- task_status 枚举兼容（review 与 backlog）
- SSR + 客户端组件 hydration 稳定性（看板与交互组件）

## 4. 测试策略（分层）

采用“静态检查 -> 单元/组件 -> 集成 -> E2E -> 数据库回归”的金字塔策略。

### 4.1 静态检查层（每次提交）
- 目标：快速发现语法、类型、基础规范问题
- 执行项：
  - eslint
  - TypeScript 类型检查
- 通过标准：无 error

### 4.2 单元测试层（快速回归）
- 目标：验证纯逻辑、状态转换、输入校验
- 重点对象：
  - 状态归一化逻辑（in_review -> review）
  - 日期与截止时间逻辑（dueSoon 计算）
  - 错误文案格式化逻辑（数据库未初始化错误）
- 建议工具：Vitest

### 4.3 组件测试层（UI 行为）
- 目标：验证关键组件交互与边界态
- 重点对象：
  - 新建任务弹窗、任务详情弹窗
  - 邀请成员弹窗、成员操作菜单
  - 设置页（资料保存、密码修改弹窗、语言切换）
- 建议工具：React Testing Library + Vitest + jsdom

### 4.4 集成测试层（前端 + Supabase 调用）
- 目标：验证页面级数据流与错误分支
- 重点对象：
  - 新建团队/项目流程
  - 项目详情页任务加载与成员加载
  - 团队详情页成员/项目展示与权限分支
- 建议工具：MSW（mock Supabase 响应）

### 4.5 端到端测试层（真实用户路径）
- 目标：覆盖最关键业务闭环
- 建议工具：Playwright
- P0 场景：
  - 注册/登录成功与失败路径
  - 创建团队 -> 创建项目 -> 创建任务 -> 拖拽到 done
  - owner 邀请成员并调整角色
  - 非管理员用户尝试受限操作（应被拒绝）
  - 设置页修改姓名、语言切换

### 4.6 数据库与权限回归层
- 目标：保证 SQL 迁移后的权限和枚举稳定
- 执行项：
  - 逐条验证 scripts/001-010 执行后 schema 正常
  - 验证 team_members RLS 无递归且权限正确
  - 验证 task_status 含 backlog/review 且历史值可归一
- 建议方式：
  - SQL smoke 脚本（必做）
  - pgTAP（可选增强）

## 5. 测试环境与数据

### 5.1 环境
- 本地开发环境（必备）
- 预发布环境（建议）
- 浏览器矩阵：Chrome、Edge（首批）；Safari（次批）

### 5.2 测试数据准备
- 用户：
  - owner 用户 1 个
  - admin 用户 1 个
  - member 用户 1 个
  - 非成员用户 1 个
- 业务数据：
  - 团队 2 个（跨团队隔离验证）
  - 每个团队至少 1 个项目
  - 每个项目至少 8 个任务（覆盖所有 status）

## 6. 用例矩阵（优先级）

### 6.1 P0（阻断发布）
- 认证：登录/注册/退出主链路
- 会话：刷新页面后会话持续有效，未登录访问受限页面处理正确
- 团队：创建团队成功，owner 自动入组
- 项目：owner/admin 可创建，member 不可创建
- 任务：创建、编辑、删除、跨列拖拽、同列排序
- 权限：非团队成员不可访问团队/项目详情
- RLS：成员不可越权操作其他团队数据

### 6.2 P1（高优先）
- 成员管理：邀请、改角色、移除成员
- 设置：修改姓名、修改密码
- 国际化：中英切换后文案与日期格式正确
- 数据异常分支：Supabase 返回错误时的提示与回退

### 6.3 P2（常规）
- 图表展示与空态文案
- 视觉与交互一致性检查
- 跨浏览器兼容细节

## 7. 进入/退出准则

### 7.1 进入准则
- SQL 迁移已执行并通过基本 smoke
- 关键测试账号已准备
- 代码已通过 lint 与 typecheck

### 7.2 退出准则
- P0 用例 100% 通过
- P1 用例通过率 >= 95%
- 无未处理的 P0/P1 缺陷
- E2E 冒烟在预发布环境连续 2 次通过

## 8. 缺陷分级与响应

- Sev-1：核心流程不可用/数据越权/数据丢失
  - 响应：立即修复，阻断发布
- Sev-2：关键功能异常但有替代路径
  - 响应：当日修复或发布前修复
- Sev-3：一般功能或展示问题
  - 响应：排期修复

## 9. 迭代排期建议（两周）

- 第 1-2 天：补齐测试框架与基础脚手架（Vitest、RTL、Playwright、MSW）
- 第 3-5 天：完成 P0 单元/组件测试
- 第 6-8 天：完成 P0 E2E 与 SQL smoke
- 第 9-10 天：补齐 P1 场景、缺陷回归、发布报告

## 10. CI 落地建议

建议在 CI 中分三段执行：
1. 静态检查：lint + typecheck
2. 快速自动化：unit + component
3. 发布门禁：e2e smoke

建议新增脚本（后续实施时加入 package.json）：
- test:unit
- test:component
- test:e2e
- test:smoke:db

## 11. 首批最小可执行清单（本周）

- 完成 10 条 P0 E2E（认证、团队、项目、任务拖拽、权限）
- 完成 15 条单元/组件测试（表单校验、错误分支、状态变更）
- 完成 1 套 SQL smoke（RLS 与枚举）
- 每次 PR 至少运行 lint + unit
